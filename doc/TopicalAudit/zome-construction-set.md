## Summary

ZomeTool-style construction system based on the `rigidbuild` library (`Zome.h`). Nodes are 62-slot connectors derived from the 31 symmetry-unique axes of an icosahedron (6 vertex + 10 face-normal + 15 edge-midpoint). Three connector families map to the three axis classes with distinct rotational symmetry orders (5, 3, 2) and cross-section shapes (pentagon, triangle, rectangle). Struts come in 3 golden-ratio lengths per family (1, φ, φ²), giving 9 strut types. The 60 rotational symmetries of the icosahedral group are generated and attached to the node type for symmetry-aware operations. The interactive editor (`test_Zome.cpp`) supports ray-picking, discrete angular shell grow via mouse wheel, and two-node connection checking.

## Sub-topics

### Icosahedral Node Geometry

The icosahedron has 12 vertices, 20 faces, 30 edges. Opposite pairs collapse to 6+10+15=31 unique axes, each yielding 2 slots (±direction) = 62 slots total.

- **Vertex axes** (6 pairs, 12 slots): 5-fold rotational symmetry → red pentagonal connectors
- **Face-normal axes** (10 pairs, 20 slots): 3-fold rotational symmetry → yellow triangular connectors
- **Edge-midpoint axes** (15 pairs, 30 slots): 2-fold rotational symmetry → blue rectangular connectors

Slot layout in `NodeTypeBuilder`: `addAxisPair(connector, family, axisIndex, orbit, axis, upHint)` creates two opposite slots with `makeFrame(axis, upHint)` orienting the local frame.

### Discrete Angular Shell Grow

Key interaction feature. When a slot is selected, `computeLayers()` computes the angle between the selected slot's world direction and every other free slot's direction. These angles are **discrete** — determined by icosahedral symmetry, not arbitrary increments. For a fresh node, ~18 distinct angles emerge: 20.9°, 31.7°, 36.0°, 54.7°, 58.3°, 60.0°, 69.1°, 72.0°, 90.0°, 108.0°, 110.9°, 120.0°, 121.7°, 125.3°, 144.0°, 148.3°, 159.1°, 180.0°.

Mouse wheel cycles `symLayer` (0 = selected slot only, N = all slots at the N-th discrete angle). Only struts **on** the shell (at exactly that angle) are previewed and grown — not everything inside the cone. This preserves directionality: the selected strut's orientation matters, not just its family.

`updatePreview()` and `growCone()` filter slots by `fabs(ang - currentAngle) < 1e-4`.

### Two-Node Connection Check

`checkConnection()` tries all 9 strut types (3 families × 3 lengths) from all free slots on `selectedNode`, checking if `previewGrow()` produces a target position matching `selectedNode2` (within 0.01 tolerance). If found, `canConnect = true` and `connectCandidate` stores the valid request. `connectTwoNodes()` then calls `construction.connectExisting()` to create the edge without creating a duplicate node.

Visual feedback: green line between nodes if connectable, red if not.

### Picking

- **Nodes**: `raySphere()` intersection with 0.25-radius spheres
- **Edges**: `rayPointDistance2()` to 8 sample points along each edge segment
- **Slots**: `rayPointDistance2()` to 6 sample points along each slot arrow (0.4 length)
- **Ray origin**: pushed 100 units behind camera along `-cam.rot.c` to avoid missing close objects when ray origin is inside the scene

### Symmetry Generation

60 rotational symmetries of the icosahedral group (order 60) are generated by:
- 6 vertex-axis rotations: 4 non-trivial rotations each (72°, 144°, 216°, 288°) → 24 ops
- 10 face-axis rotations: 2 non-trivial rotations each (120°, 240°) → 20 ops
- 15 edge-axis rotations: 1 non-trivial rotation each (180°) → 15 ops
- Plus identity → 60 total

Deduplication by exact matrix comparison (1e-10 tolerance). `makeSymmetryOp()` maps the rotation to slot-to-slot correspondences.

### Rendering

- **Nodes**: spheres (`Draw3D::drawSphere_oct`), color-coded by selection state
- **Edges**: polygonal prisms (`drawPrism`) with family-specific cross-section: pentagon (red/5-fold), triangle (yellow/3-fold), rectangle (blue/2-fold)
- **Slot arrows**: `Draw3D::drawArrow` in family colors, dimmed for non-selected/hovered
- **Preview**: blue lines + small spheres for previewed strut endpoints; cyan cone outline at `currentAngle`
- **Connection indicator**: green/red line between two selected nodes

## Parity Status

- **`Zome.h` library ↔ `test_Zome.cpp` editor**: Editor is the primary consumer of the library. All library features (grow, previewGrow, connectExisting, symmetry) are exercised.
- **No other implementations** — this is a C++-only system with no Python or JS port.

## Open Issues

- `growCone()` resets `symLayer` and `selectedSlot` after growing — user must re-select to grow again
- `computeLayers()` is called only on slot selection; if slots become occupied (by growing), layers are stale until re-selection
- No undo/redo
- No save/load of constructions
- No export to mesh or graph format
- Symmetry ops are generated but not directly used for grow — shell grow uses geometric angle comparison instead
- `connectTwoNodes()` searches for matching slot by position only, not by slot ID — could fail if multiple slots are very close
- No visual feedback for which discrete angle layer is active (only console printf)
